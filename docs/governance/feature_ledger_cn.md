# AiUE 功能台账（中文版）

这份台账用来回答三个比路线图更实际的问题：

1. 现在仓库里到底有哪些能力。
2. 哪些已经稳定，哪些还只是实验或规划。
3. 下一次讨论时，哪些东西已经做了，哪些还没做。

## 结论先行

当前最合适的形态不是 `SQLite 真相源`，而是：

- 一份面向人阅读的中文台账
- 一份面向机器读取的 JSON 台账

原因很简单：

- 现在更需要可 diff、可 code review、可跟随 commit 演进的真相源。
- 现在还没到必须用数据库管理台账的复杂度。
- 如果以后真的需要复杂查询，可以从 JSON 派生生成 SQLite，不必反过来。

对应的机器真相源是：

- [feature_ledger_cn.v1.json](C:/AiUE/docs/governance/feature_ledger_cn.v1.json)

## 状态含义

| 状态 | 含义 |
| --- | --- |
| `稳定` | 已完成并进入当前可信主线或常规回归，可作为当前能力判断依据 |
| `实验` | 已有实现和证据，但仍主要服务探索，不宜当成最高权威 |
| `开发中` | 已经开始落地，但尚未形成稳定 checkpoint 或真实通过标准 |
| `规划` | 方向与边界已明确，但尚未开始正式实现 |
| `暂停` | 确认有价值，但当前不投入主线资源 |
| `归档` | 历史能力或历史路线，保留记录，但不再作为 active line |

## 新增字段

为了让台账不只是“写下来”，而是真的能支持“先记录、后安排”，现在新增两组字段：

| 字段 | 说明 |
| --- | --- |
| `priority` | `P0 / P1 / P2 / P3 / 未知` |
| `triage_state` | `待分诊 / 已分诊 / 进行中 / 冻结 / 不适用` |

其中最关键的是：

- `priority = 未知`
- `triage_state = 待分诊`

这两个值允许我们把一个真实想法先录入台账，而不是被迫在录入当下就定优先级。

也就是说，台账现在已经支持：

- `先记下来`
- `先说明它属于哪条线`
- `先说明依赖谁`
- `之后再正式决定何时做`

## 是否要做成 SQLite

当前结论：`先不要`

建议规则是：

- `Markdown + JSON` 作为主真相源
- `SQLite` 只作为未来派生缓存或查询层

只有在下面这些条件出现时，再认真引入 SQLite：

1. 需要跨很多轮历史做复杂查询。
2. `T2` 需要很快地做筛选、排序、聚合，而 JSON 读取明显开始吃力。
3. 需要多种衍生视图，但不想把逻辑硬编码在 `T1/T2` 里。

也就是说：

`SQLite 可以有，但不应该先有。`

## 本轮台账测试结论

这次已经用一个真实的新任务验证了台账能力：

- `Q5A 黄边边缘压埋检测`

它目前被记录为：

- 所属线：`验证与质量线`
- 状态：`规划`
- 优先级：`未知`
- 分诊状态：`待分诊`
- 依赖：`Q5A / Q5C`

这说明台账现在已经能处理：

- 一个任务现在值得记
- 但优先级还不确定
- 也不必立刻承诺下一轮实现

## 我建议的使用方式

以后讨论下一步时，不要只说路线图。
先看台账，再问三个问题：

1. 这件事属于哪条线？
2. 当前状态是 `稳定 / 实验 / 开发中 / 规划 / 暂停 / 归档` 的哪一种？
3. 我们是要：
   - 把一个 `实验` 提升为 `稳定`
   - 把一个 `开发中` 收口
   - 还是开启一个新的 `规划`

这样就不会总是陷入“方向很多，但仓库现状不清楚”的状态。
